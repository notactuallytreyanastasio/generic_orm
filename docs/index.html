Chapter 1

The problem nobody wants to solve

Imagine you maintain a library that ships in 6 languages. Maybe it's an SDK. Maybe it's a database driver. Maybe it's an ORM. You find a SQL injection bug on a Tuesday afternoon.

Now multiply everything by 6. You need 6 patches, written in 6 languages, by people who know those 6 ecosystems. You need 6 PRs, 6 code reviews, 6 CI runs, 6 releases, pushed to 6 package registries. And you need them all to behave identically, because if the Python fix escapes quotes differently than the Java fix, you've just traded one vulnerability for a new inconsistency.

This is not hypothetical. AWS maintains 300+ services across 12 languages. Their Java SDK v1 to v2 migration took 7 years. Three major SDKs hit end-of-support in 2025 — zero security patches after cutoff.

The standard answer is "hire more people" or "use code generation." But code generation only handles API shapes — the actual logic, the escaping, the validation, the edge cases — those still need per-language implementations. And per-language implementations drift.

We wanted to know: what if the security logic itself was written once and compiled to every target?

Chapter 2

Temper: the language that compiles to other languages

Temper is a typed language that compiles to idiomatic output in JavaScript, Python, Rust, Java, Lua, and C#. Not transpilation. Not wrappers. Native, importable libraries that look like they were written by a developer in that language.

The idea was simple: build a security-critical library — a SQL ORM — in Temper. If the type system can prevent SQL injection at compile time in the Temper source, then the compiled output inherits those guarantees in every target language. One set of safety invariants, enforced everywhere.

Chapter 3

Five layers that make injection impossible

The ORM doesn't prevent SQL injection by being careful. It prevents it by making unsafe SQL impossible to construct. The type system won't let you. There are five layers, and they build on each other.

Layer 1 is SafeIdentifier. Every table name and column name must pass through a validator that accepts only [a-zA-Z_][a-zA-Z0-9_]*. The implementation class is sealed and not exported — there is no way to construct a valid identifier without going through validation. Try to use users; DROP TABLE as a table name and you get a compile-time error path, not a runtime surprise.

safeIdentifier("users")       // accepted
safeIdentifier("users; DROP") // bubble (rejected at construction)

Layer 2 is the SqlPart sealed hierarchy. User values never touch the "safe" append path. Instead, each data type — strings, integers, booleans, floats, dates — gets its own class that knows how to render itself safely. SqlString escapes quotes. SqlInt32 renders bare integers. SqlFloat64 renders NULL for NaN and Infinity. The interface is sealed, so the compiler enforces exhaustive handling. You cannot add a new type without updating every rendering path.

Layer 3 is the SqlBuilder itself. It has two families of methods: appendSafe() for SQL keywords and validated identifiers, and typed append() overloads for user data. There is no appendRaw. There is no escape hatch. Structure and data are separated at the API level, not by convention.

Layer 4 is the Changeset pipeline, inspired by Elixir's Ecto. It's a sealed interface — you can only construct one through the factory function. cast() requires a list of SafeIdentifier for field whitelisting. Raw params are never exposed. Values reach SQL only through type-dispatched escaping.

Layer 5 is the query builder itself. It requires SafeIdentifier for everything structural — table names, columns, ORDER BY, JOIN targets. WHERE conditions accept only SqlFragment, never raw strings. All SQL keywords come from sealed interfaces with hardcoded strings. UpdateQuery and DeleteQuery refuse to generate SQL without a WHERE clause, preventing accidental full-table mutations.

Chapter 4

Source code you can actually read

Temper source files are .temper.md — literate markdown. The source code is a document. Code lives in indented blocks. Everything else is prose: explanations, design rationale, security notes. When you open schema.temper.md, you're not reading code with comments. You're reading a document that happens to contain code.

## SafeIdentifier
Table and column names must be validated before they can appear in SQL. The SafeIdentifier interface is sealed — the implementation class is not exported, so external code must go through validation.
export sealed interface SafeIdentifier {
  public get sqlValue(): String;
}
The validator rejects any name containing SQL metacharacters.
export let safeIdentifier(name: String): SafeIdentifier throws Bubble {
  if (name.isEmpty) { bubble() }
  // ... character-by-character validation
  new ValidatedIdentifier(name)
}

This matters for a security-critical library because the reasoning lives with the code. When a reviewer opens changeset.temper.md, they don't just see a cast() function — they see the paragraph explaining why field whitelisting is enforced here, what CWE it prevents, and why the implementation class is sealed. When an auditor opens sql_model.temper.md, each SqlPart subclass comes with its escaping rationale.

The whole library is 6 source files. You can read the entire security model in an afternoon:

Chapter 5

We found 3 bugs. We fixed all 6 languages in one commit.

After building the ORM, we ran a systematic MITRE CWE Top 25 analysis against every component, phase by phase. The analysis found 3 real issues.

The first was the most interesting. In toInsertSql() and toUpdateSql(), column names were being passed as raw String values to appendSafe(). They happened to originate from SafeIdentifier — so they were safe by construction — but the type system didn't enforce this at the call site. A future refactor could silently introduce an unvalidated path.

The fix was two lines:

// Before: raw String from map key
colNames.add(pair.key);
b.appendSafe(pair.key);

// After: SafeIdentifier from looked-up FieldDef
colNames.add(fd.name.sqlValue);
b.appendSafe(fd.name.sqlValue);

The second issue: SqlDate wrapped dates in quotes without escaping. If Date.toString() ever produced a single quote, you'd get malformed SQL. We added character-by-character escaping.

The third: SqlFloat64 rendered NaN and Infinity as literal strings — not valid SQL. We made them render as NULL.

One commit. One build. All 3 fixes propagated to JavaScript, Python, Rust, Java, Lua, and C#. We verified in the compiled output — grepped the generated code for each backend and confirmed the fixes were present. No raw pair.key in any appendSafe call. Escaping loops in SqlDate. NaN guards in SqlFloat64.

In a traditional multi-language setup, that's 6 PRs, 6 reviews, 6 releases. Here it was one afternoon.

Chapter 6

This whole thing was built by one person and an AI

Here's the part that makes people uncomfortable: this entire project — the ORM, all 6 demo applications, the CI pipeline, the MITRE security analysis, the bug fixes, the pitch decks, and this website — was built by a single developer working with Claude.

The AI didn't need to know 6 languages to secure 6 backends. It worked in Temper — one language, one type system, one set of safety invariants. The compiler did the rest.

To test whether this actually matters, we ran a comparison. We asked an AI to build an ORM in Ruby with comparable prompts and comparable effort. Then we ran the same MITRE/CWE analysis on both.

The Ruby ORM had table name injection, WHERE clause injection, ORDER BY injection, GROUP BY injection, direction injection, LIMIT type confusion, and incomplete escaping. Nine attack vectors from a few prompts. Not because the AI was bad — because Ruby doesn't stop you from interpolating strings into SQL. It's the path of least resistance, and the AI took it.

In Temper, those same mistakes are impossible to express. The type system won't compile code that passes a raw string where a SafeIdentifier is required. The AI didn't write better code because it was smarter. It wrote safer code because the language wouldn't let it do otherwise.

The best security doesn't depend on the developer — or the AI — remembering to do the right thing. It makes the wrong thing impossible to express.

This is what AI-driven development actually needs. Not smarter models. Better foundations. When the language enforces invariants, the review burden collapses from N languages to 1. Adding a 7th backend requires zero security re-audit.

Chapter 7

Six apps. Six languages. Same ORM. Zero vulnerabilities.

To prove this isn't theoretical, we built 6 identical todo-list applications — one in each target language — all using the compiled ORM for every database operation. Same functionality, same UI, same security properties. Every SELECT, INSERT, UPDATE, and DELETE flows through the ORM. The only raw SQL is DDL (CREATE TABLE) — static strings with no user input.
